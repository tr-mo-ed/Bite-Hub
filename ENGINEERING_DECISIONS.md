# Engineering Decisions

هذا الملف يمثل سجل القرارات المعمارية الأساسية في مشروع **Bite Hub**.  
الهدف منه ليس فقط توثيق ما تم بناؤه، بل توثيق **لماذا** تم اختياره، وما هي المشكلة التقنية التي يعالجها، وما أثره على الأداء والأمان وقابلية التوسع.

## قرار 1: معمارية Multi-Tenancy

### المشكلة
كان أمامنا خياران رئيسيان:

1. إنشاء نسخة مستقلة من النظام لكل مقهى أو كلية
2. إنشاء نظام مركزي واحد يخدم جميع المقاهي مع عزل البيانات برمجياً

الخيار الأول يبدو بسيطاً على المدى القصير، لكنه يخلق مشاكل تشغيلية خطيرة:

- تكرار الكود وقواعد البيانات
- صعوبة الصيانة والتحديث
- ارتفاع كلفة الاستضافة والإدارة
- صعوبة إضافة مقهى جديد
- زيادة احتمال اختلاف النسخ بين العملاء

### القرار
اعتمدنا **Logical Separation** داخل قاعدة بيانات واحدة، بحيث تكون جميع الجداول التشغيلية مرتبطة بـ `cafe_id`، ويكون المقهى هو وحدة العزل الأساسية داخل النظام.

### لماذا اخترنا هذا الحل؟
- يسمح بإضافة مقهى جديد بضغطة زر من لوحة الإدارة دون إنشاء نسخة جديدة من المشروع
- يقلل استهلاك الموارد التشغيلية مقارنة بقواعد بيانات أو تطبيقات منفصلة
- يبقي النظام قابلاً للتوسع إلى عشرات أو مئات المقاهي
- يسهل مراقبة النظام وصيانته وتحديثه
- يحافظ على مصدر واحد للحقيقة `Single Source of Truth`

### كيف انعكس ذلك في الكود؟
- تم اعتماد `Cafe` كوحدة العزل الأساسية
- تم إنشاء `CafeScopedModel` ليضيف `cafe = ForeignKey(Cafe)` لكل موديل يحتاج عزلاً
- الموديلات الأساسية مثل `Category`, `Product`, `Order`, `InventoryItem` أصبحت مرتبطة بالمقهى
- تم إنشاء `Faculty` كمستوى تنظيمي أعلى يمكن استخدامه لاحقاً لتجميع المقاهي أو ربطها بكليات

### الأثر
- سهولة التوسع
- خفض الكلفة
- تسهيل إدارة المقاهي الجديدة
- توحيد بنية البيانات والتقارير

## قرار 2: معالجة مشكلة N+1 Queries

### المشكلة
في الأنظمة المبنية على ORM مثل Django، من الشائع الوقوع في مشكلة **N+1 Queries** عند جلب بيانات مترابطة مثل:

- الطلبات مع عناصر الطلب
- عناصر الطلب مع المنتجات
- المنتجات مع التصنيفات
- المنتجات مع بيانات المقهى

إذا لم تتم معالجة ذلك، فإن كل صفحة أو API قد تتحول إلى عشرات أو مئات الاستعلامات الصغيرة، ما يؤدي إلى:

- بطء في الاستجابة
- استهلاك زائد للاتصالات وقاعدة البيانات
- انهيار الأداء تحت الضغط

### القرار
اعتمدنا تحميل العلاقات بشكل صريح داخل طبقة `selectors.py` باستخدام:

- `select_related(...)`
- `Prefetch(...)`
- فهارس `Indexes` على الحقول الحرجة

### كيف طبقنا هذا الحل؟

#### في المنتجات
دوال مثل `get_products_for_cafe(cafe_id)` و`get_active_products_for_cafe(cafe_id)` تستخدم:

- `select_related("cafe", "category")`

وهذا يمنع استعلامات إضافية عند عرض:

- اسم المقهى
- التصنيف
- بيانات المنتج المرتبطة

#### في الطلبات
دوال مثل:

- `get_order_details_for_cafe(order_id, cafe_id)`
- `get_orders_for_cafe(cafe_id)`
- `get_user_orders(user)`

تستخدم:

- `select_related("cafe", "user")`
- `Prefetch("items", queryset=OrderItem.objects.select_related("product"))`

وهذا يمنع سلسلة استعلامات متكررة عند عرض:

- بيانات الطلب
- عناصر الطلب
- المنتج داخل كل عنصر

### دور الفهارس
أضفنا فهارس على المسارات الأكثر استخداماً، مثل:

- `cafe_id`
- `(cafe, category)`
- `(cafe, is_available)`
- `(cafe, status)`
- `(cafe, created_at)`

### الأثر
- تقليل عدد الاستعلامات
- تسريع الصفحات وواجهات الـ API
- تحسين السلوك تحت الأحمال المرتفعة
- تقليل الضغط على PostgreSQL مستقبلاً

## قرار 3: أمان عزل البيانات

### المشكلة
في نظام متعدد المستأجرين، أكبر خطر هندسي هو تسرب البيانات بين المقاهي بسبب:

- Query مكتوب بشكل خاطئ
- استخدام `.all()` مباشرة في View أو Admin
- نسيان شرط `cafe_id`
- ثغرة منطقية في API أو لوحة التحكم

أي خطأ من هذا النوع قد يؤدي إلى أن يرى مدير مقهى بيانات مقهى آخر.

### القرار
اعتمدنا **عزل البيانات برمجياً** عبر ثلاث طبقات:

1. `CafeScopedModel`
2. `CafeScopedQuerySet`
3. `Selectors` كبوابة إلزامية للقراءة

### كيف طبقنا هذا الحل؟

#### أولاً: الموديلات
كل موديل يحتاج عزلاً يرث من `CafeScopedModel`، وبالتالي يحمل `cafe_id` بشكل موحد.

#### ثانياً: Query API موحد
أنشأنا `CafeScopedQuerySet` وفيه:

- `for_cafe(cafe_id)`
- `for_cafes(...)`

بذلك أصبحت صيغة العزل موحدة ومركزية، بدل تكرار `filter(cafe_id=...)` عشوائياً في كل مكان.

#### ثالثاً: Selectors
أنشأنا طبقة `selectors.py` لتكون البوابة الأساسية لجلب البيانات:

- `get_active_products_for_cafe(cafe_id)`
- `get_categories_for_cafe(cafe_id)`
- `get_order_details_for_cafe(order_id, cafe_id)`
- `get_orders_for_cafe(cafe_id)`

هذا يمنع الـ Views والـ Admin من كتابة استعلامات حرة وغير مضبوطة.

### خطة التحصين المستقبلية: PostgreSQL Row-Level Security (RLS)
العزل البرمجي جيد، لكنه ليس كافياً وحده في الأنظمة الحساسة.  
لذلك قررنا أن تكون الخطوة القادمة بعد الانتقال إلى PostgreSQL هي تفعيل **RLS** على الجداول التابعة للمقهى.

الفكرة:

- كل جدول معزول يفعّل عليه `ENABLE ROW LEVEL SECURITY`
- توضع Policy تسمح فقط بالصفوف التي تطابق `current_setting('app.current_cafe_id')`
- يضبط التطبيق هذه القيمة مع بداية كل Request بعد المصادقة

### لماذا لم نطبقه الآن؟
- المشروع ما زال على SQLite في هذه المرحلة
- RLS ميزة خاصة بـ PostgreSQL
- تطبيقه الصحيح يتطلب:
  - PostgreSQL فعلي
  - إدارة Session Variables
  - إعداد connection pooling بشكل منضبط

### الأثر
- حماية أقوى ضد أخطاء الاستعلام
- تقليل خطر تسرب البيانات بين المقاهي
- رفع المستوى الأمني للنظام إلى طبقتين:
  - طبقة التطبيق
  - طبقة قاعدة البيانات

## قرار 4: تحمل الضغط والـ High Concurrency

### المشكلة
في أوقات الذروة، مثل فترات الاستراحة أو أوقات التجمعات الطلابية، قد يرسل مئات أو آلاف المستخدمين الطلبات نفسها تقريباً في نفس اللحظة:

- جلب قائمة المقاهي
- جلب قائمة المنتجات
- جلب الأقسام
- تصفح الواجهة الرئيسية

إذا تم توجيه كل هذه الطلبات مباشرة إلى قاعدة البيانات، فسيرتفع الحمل بسرعة، حتى لو كانت الاستعلامات نفسها متكررة.

### القرار
اعتمدنا مبدئياً هيكلية جاهزة لإدخال **Redis Caching** كجزء من معمارية الأداء العالي.

### الفكرة
البيانات التي تتغير بمعدل أقل من الطلبات، مثل:

- المنتجات المتاحة
- التصنيفات
- المقاهي النشطة

يجب ألا تُقرأ من قاعدة البيانات في كل مرة.  
بدلاً من ذلك، سيتم تخزينها مؤقتاً في Redis، ثم:

- أي طلب متكرر يقرأ من الكاش
- وعند تعديل المنتج أو الفئة أو حالة التوفر، يتم فقط إبطال الكاش وإعادة بنائه

### لماذا Redis؟
- سريع جداً في القراءة
- مناسب للبيانات المؤقتة والمتكررة
- يقلل الضغط على PostgreSQL بشكل كبير
- مناسب لسيناريوهات الذروة ذات القراءة العالية

### الأثر المتوقع
الهدف المعماري هنا هو تقليل الحمل على قاعدة البيانات بنسبة كبيرة في المسارات الأكثر تكراراً، وقد يصل ذلك عملياً إلى **تقليل يصل إلى 90%** في الاستعلامات القرائية المتكررة وقت الذروة، إذا تم تطبيق:

- Redis Cache
- invalidation صحيح
- selectors ثابتة
- API paths واضحة

### ملاحظات التنفيذ
هذا القرار موثق الآن كجزء من الخطة القريبة، وسيتم تنفيذه مع:

- PostgreSQL
- Docker
- إعدادات الإنتاج
- تحسين مسارات المنتجات والمقاهي أولاً

---

## خلاصة عامة

القرارات السابقة ليست تحسينات شكلية، بل تمثل الأساس الذي سيجعل Bite Hub:

- قابلاً للتوسع
- آمناً في عزل البيانات
- أكثر كفاءة تحت الضغط
- أسهل في الصيانة والتقييم

النهج الذي اعتمدناه هو:

- عزل برمجي صارم عبر `cafe_id`
- منع N+1 على مستوى الـ selectors
- تجهيز الطريق لـ PostgreSQL + RLS + Redis
- إبقاء الـ Views خفيفة والـ Business Logic داخل `services.py`

هذا هو الاتجاه الصحيح لنظام متعدد المقاهي يحتاج أن ينمو بدون أن ينهار تحت التعقيد.

## قرار 5: الانتقال إلى ASGI و Django Channels

### المشكلة
النظام الحالي مبني عملياً على نمط طلب/استجابة تقليدي، وهذا مناسب للعمليات العادية، لكنه غير مثالي لسيناريوهات تحتاج تحديثاً فورياً مثل:

- تحديث حالة الطلب لحظة بلحظة
- إشعارات فورية للمدير والعميل
- لوحات متابعة حية
- تقليل الاعتماد على polling المتكرر من الواجهة

الاعتماد على `polling` فقط في هذه الحالات يسبب:

- طلبات HTTP كثيرة ومتكررة
- حمل زائد غير ضروري على الخادم
- تأخير في ظهور التحديثات
- استهلاك أعلى للموارد مع زيادة عدد المستخدمين

### القرار
سننتقل إلى **ASGI** مع **Django Channels** كطبقة التشغيل المعتمدة للميزات الحية `Real-time`.

### لماذا اخترنا هذا الحل؟
- يدعم WebSockets بشكل منظم داخل Django
- يتيح بث الإشعارات وتحديثات الطلبات دون إعادة استعلام مستمرة
- أكثر كفاءة من polling عند وجود آلاف الاتصالات المتزامنة
- يفتح الطريق لميزات حية مستقبلية مثل:
  - live dashboards
  - live order queue
  - in-app notifications

### الدور المعماري المتوقع
- `ASGI` سيكون طبقة التشغيل الأساسية بدلاً من الاعتماد على WSGI فقط
- `Channels` سيدير:
  - الاتصالات الحية
  - مجموعات المقاهي `cafe groups`
  - بث التحديثات حسب `cafe_id`
- `Redis` سيستخدم أيضاً كـ `channel layer`

### الأثر
- تقليل الاعتماد على polling
- تحسين تجربة المستخدم في الطلبات والإشعارات
- تجهيز النظام لمستوى تشغيل أقرب إلى Enterprise SaaS

## قرار 6: نظام تتبع الأخطاء باستخدام Sentry

### المشكلة
عند توسع النظام إلى عشرات المقاهي وآلاف المستخدمين، يصبح تتبع الأخطاء يدوياً عبر السجلات النصية غير كافٍ.  
المشكلة ليست فقط معرفة أن خطأ حدث، بل معرفة:

- أين حدث؟
- في أي سطر؟
- في أي endpoint؟
- مع أي مستخدم؟
- وفي سياق أي مقهى `cafe_id`؟

بدون هذه الرؤية، يصبح إصلاح الأعطال أبطأ وأكثر كلفة.

### القرار
سنعتمد **Sentry** كأداة مركزية لتتبع الأخطاء في:

- Backend (Django)
- Frontend (Flutter / لاحقاً Web إن وجد)

### لماذا اخترنا هذا الحل؟
- يعطي stack trace دقيقاً
- يربط الخطأ بالسياق التنفيذي
- يدعم التنبيهات الفورية
- مناسب للإنتاج وفرق التشغيل

### ما الذي سنربطه داخل Sentry؟
- `cafe_id`
- `user_id`
- نوع البيئة `env`
- اسم المسار أو الـ endpoint
- معلومات الإصدار `release/version`

### الأثر
- اكتشاف أسرع للأعطال
- تقليل وقت التشخيص
- رفع جودة التشغيل والدعم الفني

---

## قرار 7: معالجة الصور والتخزين الخارجي

### المشكلة
الاعتماد على التخزين المحلي `Local Storage` للصور داخل الخادم ليس مناسباً لنظام SaaS متوسع، للأسباب التالية:

- يجعل التوسعة الأفقية أصعب
- يزيد حساسية الخادم للملفات والنسخ الاحتياطي
- قد يبطئ الاستجابة إذا تم تحميل أو معالجة الصور داخل مسار الطلب المباشر
- يجعل نقل النظام بين البيئات أكثر تعقيداً

### القرار
سنعتمد:

- **S3-Compatible Object Storage** للصور والملفات
- **Celery** لتنفيذ المهام الخلفية
- **Redis** كـ broker / backend للمهام

### لماذا اخترنا هذا الحل؟
- Object Storage مناسب للصور والوسائط على المدى الطويل
- يفصل التخزين عن الخادم التطبيقي
- Celery يسمح بتنفيذ المعالجة الثقيلة خارج مسار الطلب
- تحسين الصور أو توليد النسخ المصغرة لن يعطل الـ API

### ما الذي سيُنقل إلى الخلفية؟
- رفع الصور النهائية
- توليد thumbnails
- ضغط الصور وتحسينها
- حذف الملفات القديمة أو تنظيف النسخ غير المستخدمة

### الأثر
- API أسرع
- خادم التطبيقات أخف
- بنية أكثر جاهزية للتوسع

---

## قرار 8: نظافة الـ API والتوثيق الرسمي

### المشكلة
في الأنظمة التي تنمو بسرعة، تظهر مخاطر متكررة:

- endpoints غير موثقة
- endpoints قديمة وغير مستخدمة
- تضارب بين ما تتوقعه الواجهة وما يرسله الخادم
- تراكم مسارات غير RESTful أو غير متسقة

هذا يخلق عبئاً كبيراً على التطوير والصيانة والتقييم.

### القرار
سنعتمد **drf-spectacular** لتوليد:

- OpenAPI schema
- Swagger UI
- Redoc

وسنستخدمه كمرجع رسمي لعقود الـ API.

### لماذا اخترنا هذا الحل؟
- يجعل التوثيق مرتبطاً فعلياً بالكود
- يكشف المسارات غير المستخدمة أو غير المكتملة
- يدعم التقييم الفني والاختبار والتكامل
- يساعد على فرض معايير RESTful واضحة

### ما الذي سنفرضه لاحقاً؟
- توحيد أسماء المسارات والموارد
- توحيد رموز الحالة `status codes`
- توحيد الـ serializers الخاصة بالإنشاء/العرض/التحديث عند الحاجة
- مراجعة كل endpoint قديم أو مكرر

### الأثر
- API أوضح
- تقليل الالتباس بين الفرق
- تسهيل التكامل مع الواجهات والأدوات الخارجية

## قرار 9: ربط دورة حياة الطلب بالكامل بالـ WebSockets

### المشكلة
إرسال إشعار عند إنشاء الطلب فقط لا يكفي لتشغيل لوحة مقهى حية. إذا بقيت تحديثات الحالة مثل `ACCEPTED` و`PREPARING` و`READY` و`COMPLETED` محصورة داخل HTTP فقط، فستبقى الواجهة بحاجة إلى polling أو ستعرض حالة قديمة.

### القرار
تم ربط دورة حياة الطلب بالكامل بطبقة WebSocket المعزولة لكل مقهى:

- عند إنشاء الطلب: إرسال `order.created`
- عند تحديث حالة الطلب: إرسال `order.updated`
- كل مقهى يستقبل فقط عبر المجموعة الخاصة به: `cafe_orders_<cafe_id>`

### كيف نُفذ ذلك؟
- تم نقل تغيير حالة الطلب إلى `update_order_status(...)` داخل طبقة `services`
- تم فرض التحقق من انتقال الحالة داخل الخدمة نفسها بدل تركه موزعاً في الـ views
- بعد نجاح الحفظ في قاعدة البيانات يتم استدعاء `_broadcast_order_event(...)`
- الـ consumer يمرر الحدث والـ payload كما هما إلى عميل WebSocket

### الأثر
- تحديثات حية متسقة مع قواعد العمل
- إزالة التكرار بين مسارات إنشاء الطلب وتحديث حالته
- تمهيد مباشر لشاشات إدارة الطلبات الحية في Flutter بدون polling مستمر
